1. Executive Overview 
1.1 Platform Vision 
Futurestick is a comprehensive, AI-powered career and recruitment ecosystem engineered to unify the 
entire hiring lifecycle in a single platform. Unlike traditional job boards that act purely as listing 
aggregators, Futurestick combines a professional social network, an advanced AI-driven career toolkit, a 
fully integrated live video interview engine with proctoring, and a competitive hiring environment 
featuring contests with prizes. The platform is designed to serve three primary actor types — job seekers 
(users), employers (companies), and platform administrators — each with dedicated panels, distinct role-
based access, and tailored workflows. 
The system is architected as a polyglot, microservices-oriented application. The frontend uses React with 
Tailwind CSS for the application panels, while the public landing page is built using vanilla HTML, CSS, and 
JavaScript for maximum performance and SEO. The backend uses a hybrid runtime: Python (FastAPI) 
handles all AI-intensive operations such as resume building, job matching, career counseling, and live 
interview proctoring; Node.js (Express + Socket.IO) handles real-time features such as chat, notifications, 
video signaling, and the social feed. Persistence is split across MySQL for transactional, relational data 
(users, companies, jobs, applications, payments) and MongoDB for flexible, document-shaped data (social 
posts, feeds, chat messages, interview recordings metadata, AI artifacts). 
1.2 Document Scope 
This document is a complete technical blueprint. It does not contain a business plan, marketing strategy, 
financial projections, or revenue model. Instead, it provides exhaustive coverage of: 
• All user-facing pages and their components across all three panels. 
• Complete frontend architecture, routing, state management, and component hierarchy. 
• Complete backend architecture, including Python and Node.js service responsibilities, API endpoints, 
and inter-service communication. 
• Complete database schema for both MySQL (relational) and MongoDB (document) stores, including 
indices and relationships. 
• All AI modules with their model choices, pipelines, prompts, and integration patterns. 
• The live video interview engine including face tracking, eye tracking, tab-switch detection, multi-
person detection, and other proctoring features. 
• Security model, role-based access control, authentication, and authorization. 
• Real-time communication architecture (chat, notifications, video signaling). 
• Deployment architecture, infrastructure, CI/CD, monitoring, and scaling strategy. 
1.3 Three-Panel Architecture Summary

Futurestick is organized into three distinct yet interconnected panels, each rendered as a separate React 
application sharing a common component library and design system: 
• User Panel: The job-seeker experience. Contains the social feed, job discovery, AI tools (resume 
builder, career counselor, interviewer, job matcher, auto-apply), profile, applications, interviews, 
chat, connections, and learning paths. 
• Company Panel: The employer experience. Contains job posting, application management, 
candidate pipeline, interview scheduling, live interview hosting, contest creation, hiring analytics, 
company social posts, and HR team management. 
• Admin Panel: The platform operator experience. Contains user moderation, company verification, 
content moderation, payment management, AI usage analytics, system health, role management, 
and platform-wide configuration. 
Role-based access control (RBAC) is enforced at three layers: at the API gateway via JWT claims, at the 
service layer via decorators on each route, and at the database layer via row-level access policies. Every 
endpoint declares its required role(s) and permissions, and the frontend dynamically renders only what 
the current user is authorized to see.

2. System Architecture 
2.1 High-Level Architecture 
Futurestick uses a layered, service-oriented architecture composed of seven primary tiers. Requests 
originate from one of four client surfaces — the public landing page, the user panel, the company panel, 
or the admin panel — and flow through a CDN-fronted load balancer into the API gateway. The gateway 
authenticates the request, applies rate limiting and request shaping, and routes the call to the appropriate 
backend service. Backend services are split between Node.js (real-time, transactional, social) and Python 
(AI, computer vision, machine learning). Both runtimes share the data layer, which is a hybrid of MySQL, 
MongoDB, Redis (caching and pub/sub), and an object store (S3-compatible) for media. 
2.1.1 Tier-by-Tier Breakdown 
• Tier 1 — Client: Four distinct client applications: the marketing/landing site (HTML/CSS/JS), and 
three React + Tailwind single-page applications for users, companies, and admins. Mobile apps can 
later wrap these via React Native or be served as PWAs. 
• Tier 2 — Edge / CDN: CloudFront or Cloudflare for static assets, image transformations, and DDoS 
mitigation. The landing site is fully static and served from the edge. The React apps are also edge-
cached but hydrate against APIs. 
• Tier 3 — API Gateway: An NGINX or Kong-based gateway that terminates TLS, validates JWT tokens, 
applies rate limiting, performs request logging, and routes traffic. The gateway also handles 
WebSocket upgrades for real-time channels. 
• Tier 4 — Service Layer (Node.js): Auth service, user service, company service, job service, 
application service, social/feed service, chat service, notification service, interview signaling service, 
payment service. Built with Express and Socket.IO. 
• Tier 5 — Service Layer (Python): AI orchestration service, resume builder, resume enhancer, AI 
interviewer, AI career counselor, job matcher, auto-apply engine, and the live interview proctoring 
engine (face tracking, eye tracking, multi-person detection). Built with FastAPI, Celery, and 
PyTorch/MediaPipe/OpenCV. 
• Tier 6 — Data Layer: MySQL 8 (transactional), MongoDB 7 (document), Redis 7 (cache + pub/sub + 
sessions), Elasticsearch (search), S3 (media). 
• Tier 7 — Background Workers: Celery (Python) workers for AI tasks; BullMQ (Node) workers for 
emails, notifications, scheduled jobs, and webhooks. 
2.2 Why a Hybrid Backend? 
A single backend runtime would force compromises. Node.js excels at concurrent I/O-bound workloads 
— chat fan-out, WebRTC signaling, social feed delivery, real-time notifications — but is suboptimal for 
CPU-bound machine learning. Python has the richest AI ecosystem (PyTorch, Hugging Face, MediaPipe,

OpenCV, LangChain) but is slower for high-concurrency I/O. By splitting responsibilities along workload 
type, Futurestick can scale each tier independently and use the best tool for each job. 
Inter-service communication uses two patterns. For synchronous calls (e.g., the user service requesting a 
resume score from the AI service), services communicate over internal HTTP with mTLS. For asynchronous 
workflows (e.g., enqueuing an AI resume generation job, or fanning out a new social post to followers), 
services use Redis Streams or RabbitMQ as the message bus. 
2.3 Why a Hybrid Database? 
Relational integrity matters for entities like users, companies, jobs, applications, payments, and interview 
schedules. These have strict schemas, foreign-key relationships, and require ACID guarantees — MySQL 
is the right fit. However, many features have variable, evolving, or hierarchical shapes: social posts with 
arbitrary embedded media, comments and nested replies, chat message threads, AI interaction history, 
generated resume documents, interview event logs, and feed personalization data. Forcing these into 
rigid tables leads to schema sprawl and join-heavy queries. MongoDB stores them as documents naturally. 
The two databases are not duplicated; they are complementary. A user's identity and authentication live 
in MySQL; their social feed posts and chat history live in MongoDB. Cross-store consistency is maintained 
via an outbox pattern: writes to MySQL emit domain events to a Kafka or Redis stream, which a worker 
consumes and applies to MongoDB.

4. Frontend Architecture 
4.1 Application Topology 
Futurestick's frontend is split into four independent deployable units, each tuned for its purpose. The 
landing page is a static HTML/CSS/JS site optimized for first-paint and SEO. The three React panels (user, 
company, admin) share a common design system and component library, but are deployed as separate 
single-page applications behind subdomains: app.futurestick.com (user), business.futurestick.com 
(company), and admin.futurestick.com (admin). Each panel boots independently, ships only the code its 
actor needs, and reaches the same backend API. 
4.1.1 Why Separate SPAs Instead of One App? 
• Bundle Size: Admins do not need the social feed code; users do not need the candidate-pipeline 
code. Splitting cuts each bundle by 40-60%. 
• Security Posture: Admin code never ships to user browsers, reducing the surface area for inspection 
and reverse-engineering of moderation logic. 
• Independent Deployment: Each panel can be released, rolled back, or A/B tested in isolation. 
• Routing Clarity: Each domain has its own auth flow and route tree, preventing accidental cross-
panel link leakage. 
4.2 Shared Design System 
All three panels consume a shared package, @futurestick/ui, which exports atomic components (Button, 
Input, Modal, Card, Avatar, Badge, Toast), composite components (DataTable, FormBuilder, FilePicker, 
RichTextEditor, VideoTile, Chart), and design tokens (colors, spacing, typography, shadows, motion). The 
package is built with Tailwind's preset feature so each consumer extends it; tokens are defined in a single 
tailwind.preset.js file. The library uses Storybook for component documentation and ships its own visual 
regression test suite using Chromatic.

4.5 State Management Strategy 
State in Futurestick is divided into four categories, each handled by the most appropriate tool. Server 
state is cached in TanStack Query, which deduplicates requests, refetches on window focus, and provides 
optimistic update primitives. Client UI state (modals open, sidebar collapsed, theme, draft messages) lives 
in Zustand stores split by feature slice. Form state is owned by React Hook Form, with Zod schemas used 
for validation and TypeScript inference. URL state — filters, pagination cursors, current tab — lives in the 
URL via search params, making every view shareable and bookmarkable.

4.6 Real-Time Layer 
Real-time data is delivered through Socket.IO. A single client-side socket connects on app boot after 
authentication, then joins multiple rooms based on the user's identity, conversations, and active 
interview. Events are dispatched through a typed event bus that integrates with TanStack Query's 
invalidation system, so receiving a new chat message both updates UI immediately and invalidates the 
relevant query keys.

5. User Panel — Pages and Features 
The user panel is the primary surface for job seekers. It is structured around six pillars: Social Feed, Job 
Discovery, AI Career Tools, Profile and Resume, Interviews and Applications, and Communication. Each 
pillar contains multiple pages, components, and flows. This section covers every page, the components

5.1 Onboarding & Authentication 
5.1.1 Sign Up Page 
New users land on a multi-step onboarding flow that collects credentials, personal information, and initial 
career data so the platform can immediately personalize the experience. The flow uses React Hook Form 
with a Zod schema spanning all steps; data is persisted in session storage so refreshes do not lose progress. 
• Step 1 — Account: Email, password (with strength meter), agree to terms. Optional OAuth: Google, 
LinkedIn, GitHub. 
• Step 2 — Identity: Full name, headline, current location, phone (optional), date of birth, gender 
(optional). 
• Step 3 — Career Snapshot: Current status (student, working, looking, freelancing), years of 
experience, current role, current company, industry. 
• Step 4 — Skills & Interests: Tag-based skill picker with autosuggest from a curated 12,000-skill 
taxonomy; interest topics for feed personalization. 
• Step 5 — Resume Import: Upload existing resume (PDF/DOCX) — server runs the resume parser and 
prefills the entire profile, or skip and start blank. 
• Step 6 — Verify Email: OTP-based email verification before first login. 
5.1.2 Login Page 
Standard email/password plus OAuth providers. Includes a passwordless magic-link option for users who 
forgot their password. Login attempts are rate-limited at the API gateway: five attempts per IP per 15 
minutes, then a CAPTCHA challenge. After successful login, the server returns a short-lived access token 
(15 min) and a long-lived refresh token (30 days, HTTP-only cookie). Two-factor authentication via TOTP 
or SMS is offered after login if the account has 2FA enabled. 
5.1.3 Forgot Password Flow 
Three-step flow: enter email, receive 6-digit code, set new password. The reset token is single-use and 
expires in 15 minutes. The user is logged out of all other sessions on success.

5.5 AI Career Tools 
5.5.1 AI Career Counselor (/ai/counselor) 
A conversational coach that helps the user plan their career. The interface is a chat-first experience with 
structured side panels for action items, recommended skills, and learning paths. The counselor draws on 
the user's full profile, target roles, recent applications, and a knowledge graph of careers, skills, and

5.5.2 AI Job Matcher (/ai/matcher) 
A daily-refreshed personalized list of the top 20 jobs matched to the user's profile, with detailed score 
breakdowns and explanations. The matcher uses a hybrid approach: a sentence-transformer embedding 
of the user's profile and each job is computed; cosine similarity gives a baseline score; this is then refined 
by a learned reranker that incorporates explicit signals (location preferences, salary expectations, work 
mode) and historical click/apply behavior. Matches are explained in plain English: "Strong match because 
you have 4 years of React experience and the role requires 3+; gap on Kubernetes which is preferred but 
not required." 
5.5.3 AI Auto-Apply (/ai/auto-apply) 
A consent-based agent that, after the user defines their preferences and approves a daily quota, 
automatically applies to matched jobs above a configurable score threshold. Each application uses the 
user's tailored resume for the role (generated on-the-fly), an AI-written cover letter customized to the job 
description, and answers to screening questions inferred from the user's profile. The user reviews and 
approves each application before submission by default; advanced users can enable full autonomy with 
an audit trail. A daily summary email lists every job applied to.

8.3 Authentication & Authorization 
8.3.1 Token Strategy 
Futurestick uses a dual-token JWT model. The access token is short-lived (15 minutes), holds the user ID, 
role, scopes, and panel context, and is sent on every API call as a Bearer header. The refresh token is long-
lived (30 days), HTTP-only, secure, SameSite=strict, and never accessible to JavaScript. When the access 
token expires, the client silently calls /auth/refresh to obtain a new access token. Refresh tokens rotate 
on every use, and the previous token is invalidated; reuse of an old refresh token triggers immediate 
session revocation across all devices, indicating a likely theft. 
8.3.2 RBAC Model 
Authorization is role-based with attribute-based extensions. Each user has a global role (USER, 
COMPANY_MEMBER, ADMIN) plus context roles within companies (OWNER, ADMIN, HIRING_MANAGER, 
RECRUITER, INTERVIEWER, READ_ONLY). Permissions are expressed as resource:action tuples (e.g., 
job:create, candidate:view, interview:host). Routes declare required permissions; the auth middleware 
verifies the JWT, looks up the user's effective permission set (cached in Redis for 60 seconds), and 
authorizes or rejects. Sensitive admin actions require step-up authentication via WebAuthn at the 
moment of action. 
8.3.3 OAuth Providers 
Google, LinkedIn, GitHub, Microsoft are supported. The OAuth flow uses PKCE on the client. On callback, 
if the email matches an existing account, the OAuth identity is linked (with explicit user confirmation); 
otherwise a new account is created and prefilled from OAuth claims. LinkedIn-sourced accounts can 
additionally pull profile, experience, and education to seed the Futurestick profile.

8.4 Inter-Service Communication 
8.4.1 Synchronous (HTTP) 
Services communicate over internal HTTP using a shared SDK auto-generated from each service's OpenAPI 
spec. Calls carry the original request ID and trace headers for end-to-end tracing. Internal HTTP uses mTLS,

9. AI Modules — Detailed Specifications 
AI is woven throughout Futurestick. This section specifies each AI module: its inputs, outputs, model 
selection strategy, prompt or pipeline shape, evaluation method, latency target, and failure-handling 
approach. 
9.1 AI Orchestrator 
A central FastAPI service that fronts every AI feature. It manages prompt templates (versioned in a 
registry), selects the underlying model based on feature, language, and cost budget, applies rate limits 
and quota tracking per user and per company, captures every input and output to an evaluation store 
(with PII redaction), and exposes a unified streaming API to clients. The orchestrator supports OpenAI, 
Anthropic, Google Gemini, and self-hosted open-source models (via Ollama or vLLM) interchangeably; 
switching providers is a feature-flag change. 
9.2 Resume Parser 
Converts an uploaded resume (PDF or DOCX) into structured JSON conforming to the Futurestick profile 
schema. Pipeline: (1) text extraction with PyMuPDF and python-docx, fallback to OCR via Tesseract for 
scanned PDFs, (2) layout-aware section detection using a fine-tuned transformer that classifies each text 
block (header, contact, summary, experience, education, skills, etc.), (3) entity extraction within each 
section using few-shot LLM prompts, (4) normalization (dates, locations, skills mapped to taxonomy), (5) 
confidence scoring per field. Fields below 0.6 confidence are flagged for user review. Target latency: under 
8 seconds for a 2-page resume. 
9.3 Resume Builder 
Generates resumes from the user's structured profile and a chosen template. The template is a Jinja-
rendered HTML structure with CSS, converted to PDF by WeasyPrint. The builder also offers AI-assist 
within each section: bullet-point rewrites that strengthen action verbs, add quantification, and align with 
target keywords. The user supplies a job description (optional) and the builder uses a constrained 
generation strategy that preserves factual content (no hallucinated achievements) while improving 
phrasing. 
9.4 Resume Enhancer 
Multi-pass critique. Pass 1 — ATS analyzer (deterministic): keyword extraction, font and layout checks, 
tabular detection, file-size and image-count checks. Pass 2 — LLM critique: clarity, impact, redundancy, 
action-verb diversity, quantification gaps, with a per-bullet 0-100 score and rewrite suggestion. Pass 3 — 
Job-specific alignment (if a JD is provided): keyword match score, missing critical skills, suggested 
reordering. The output is a side-by-side diff with one-click accept/reject per change.

9.5 AI Job Matcher 
The matcher combines vector similarity with a learned reranker. 
• Stage 1 — Embeddings: Each user's profile and each job description are encoded as 768-dim 
sentence-transformer embeddings (mxbai-embed-large or similar). User embeddings are 
recomputed on profile update; job embeddings on job edit. Stored in pgvector or Pinecone. 
• Stage 2 — ANN Retrieval: For a given user, the top 500 most similar jobs are retrieved via 
approximate nearest neighbor search. 
• Stage 3 — Reranker: A gradient-boosted tree (XGBoost) reranks the 500 candidates using explicit 
features: cosine similarity, location distance, salary alignment, work-mode match, experience 
bracket match, language overlap, education match, skill-graph proximity, recency, and historical 
engagement (clicks, applies, dwell time). Trained on platform engagement data. 
• Stage 4 — Diversity: Maximal Marginal Relevance is applied to ensure the final 20 are not all from 
the same company or role. 
• Stage 5 — Explanation: An LLM generates a short, plain-English rationale for each surfaced job: 
which strengths matter and which gaps are minor. 
9.6 AI Auto-Apply 
A consent-bound agent. The user defines: target roles, locations, work mode, salary minimum, daily quota 
(default 5), and the minimum match score (default 80). A nightly Celery job retrieves matched jobs above 
threshold, generates a tailored resume per job, an AI-written cover letter, and answers screening 
questions from the user's profile. The user reviews each application in a queue at /ai/auto-apply by 
default, approving or rejecting; experienced users can enable full autonomy with mandatory daily 
summary emails. Every applied job is logged in the user's application list with an "AI-applied" badge for 
transparency. 
9.7 AI Career Counselor 
A LangGraph-orchestrated agent. It maintains a running conversation, has access to tools (profile reader, 
job searcher, salary benchmarker, course recommender, skill-graph traverser, market-trend reporter), 
and a long-term memory of the user's stated goals, constraints, and prior decisions (stored in a per-user 
vector store). The counselor uses a chain-of-thought style reasoning loop: Plan → Use Tool → Observe → 
Reflect → Respond. It refuses to make decisions for the user and always presents trade-offs.

9.12 AI Moderation

description, profile field, competition entry). Categories: spam, hate, harassment, NSFW, scam, IP

10. Live Interview Engine 
10.1 Goals 
Futurestick's built-in live interview engine eliminates the need for third-party video tools (Zoom, Google 
Meet) by combining low-latency video, role-aware tooling (whiteboard, code editor), and a robust 
proctoring layer in a single, integrated experience. The engine is engineered to feel as natural as a video 
call while continuously producing structured signal that helps interviewers evaluate candidates fairly and 
detect dishonest behavior. All proctoring is opt-in, transparent, and explained to the candidate before the 
session starts. 
10.2 Media Architecture 
The video plane is built on WebRTC with a Selective Forwarding Unit (mediasoup). For one-on-one 
interviews, the system can fall back to peer-to-peer to minimize bandwidth and latency; for panel 
interviews and recorded sessions, all media flows through the SFU. STUN servers (Google public + 
redundant) enable host discovery; coturn TURN servers enable relay through restrictive NATs. Media 
tracks are encrypted end-to-end with insertable streams; the SFU sees only opaque bytes. 
10.2.1 Signaling Flow 
1. Candidate clicks "Join" → POST /interviews/:id/start 2. interview-svc validates: time 
window, identity, consent 3. interview-svc returns: room token, SFU URL, ICE servers 4. Client 
connects to SFU via WebSocket (signaling) 5. SFU authenticates room token (HMAC-signed) 6. 
Client publishes audio/video/screen tracks 7. SFU forwards to other participants based on room 
state 8. Proctoring tracks (video frames sampled) → proctor-svc via WebSocket 9. Periodic 
event publishes (transcripts, signals) to interview-svc 10. On end → finalization job: 
transcript, recording mux, scorecard prefill 
10.3 Proctoring Subsystem 
Proctoring runs entirely client-side for sensitive vision (no raw video is sent to a proctoring server unless 
the candidate explicitly consents to recording). The browser samples webcam frames at 4-6 FPS into an 
off-screen canvas, runs lightweight models in WebGL/WASM via MediaPipe Tasks, and emits compact 
event messages (bools, angles, IDs) to the proctor service. The service aggregates events into per-minute 
summaries and a cumulative integrity score, which is shown to interviewers in real time. 
10.3.1 Proctoring Signals 
• Face Presence: Is exactly one face detected? Continuous absence beyond 3 seconds is a flag. 
• Face Identity: Periodic face-embedding comparison against the candidate's verified identity photo. 
Drift beyond a similarity threshold triggers a flag (possible candidate swap). 
• Multi-Person Detection: More than one face in frame for more than 2 seconds is a flag (possible 
coaching).

• Eye Gaze Tracking: MediaPipe Face Mesh + iris landmarks estimate gaze vector. Sustained off-
screen gaze (looking at second monitor or notes) is flagged. 
• Head Pose: Pitch/yaw/roll estimation. Repeated extreme deviations (looking down at phone, behind 
monitor) are flagged. 
• Mouth Activity vs Speech: Detects when the candidate's mouth is closed but speech is heard 
(someone else talking) or vice versa (lip-sync issues). 
• Tab Switch Detection: Page Visibility API + focus events detect when the browser tab is no longer 
active. Each occurrence and duration logged. 
• Window Blur: Window-level focus loss (alt-tab to another app). 
• Copy/Paste Activity: Clipboard events captured to detect pasting answers. 
• Network Quality: RTT, jitter, packet loss; correlated with anomalies (a glitch may explain a flagged 
event). 
• Audio Anomaly: Background voices, keyboard clatter, room echo via simple audio classification. 
• Device Sanity: Virtual camera detection (heuristics on device name, image variance). Flag if a virtual 
camera is in use. 
• Browser Sanity: DevTools open detection, headless browser markers, automation flags. 
• Environment Scan (Optional): At session start, candidate is asked to slowly pan camera around the 
room; the recording is saved as evidence. 
• Screen Share: For coding rounds, full-screen share is required and analyzed for unexpected window 
changes. 
10.3.2 Integrity Score 
Each signal contributes a weighted deduction to a starting score of 100. Weights are tunable per company 
policy. The final score is a continuous 0-100 plus a categorical recommendation (Clean, Minor, Significant, 
Severe). The score is never the sole basis for a hiring decision; it is a flag that prompts the interviewer to 
ask follow-up questions or, in extreme cases, to invalidate the session. 
10.4 Tools Inside the Room 
• Whiteboard: Excalidraw-based collaborative drawing. Multi-cursor, shape libraries, export as image. 
• Code Editor: Monaco editor with multi-language support (JS, TS, Python, Java, C++, Go, Rust, SQL, 
etc.), ESLint/Prettier integration, run button executing code in a sandboxed Docker runner via the AI 
service. Includes test-case panel for live evaluation. 
• Question Bank: Interviewer can pin pre-loaded questions from the template; per-question note-

• File Share: Drag-and-drop documents (e.g., a problem statement PDF). Auto-deleted post-interview

• Chat: Sidebar text chat for sharing links and clarifications without disrupting voice flow.

• Transcript: Live captions via Whisper streaming, with speaker diarization. Searchable, exportable. 
10.5 Recording & Storage 
Recording is opt-in and explicitly consented to by the candidate before joining. When enabled, the SFU 
server-side records each track separately at the highest quality, then post-processes: mixed audio + video 
MP4 (for review), separate per-track WebM (for reanalysis), full transcript with timestamps, proctor-event 
timeline, and AI-generated summary. Recordings are encrypted at rest with KMS, retained per company 
policy (default 90 days, configurable), and access-logged. 
10.6 AI-Generated Interview Summary 
Within minutes of the interview ending, the system delivers a structured summary to interviewers: an 
executive overview, per-question scoring with quoted candidate excerpts, identified strengths and 
concerns, suggested follow-up questions for the next round, and a sentiment/communication analysis. 
This summary populates the interviewer's scorecard as a draft, which they review and edit before 
submitting. 
10.7 Network Resilience 
Lossy networks are common in many regions. The engine adapts: bitrate is auto-reduced to as low as 100 
kbps for video and 16 kbps for audio; if video fails entirely, the call falls back to audio-only with a candidate 
avatar; if both fail, the session pauses with a buffered "reconnecting" state for up to 60 seconds before 
officially marking a disruption. All disruptions are logged and visible in the post-interview report, ensuring 
the candidate is not penalized for network issues.

11. Database Schemas 
11.1 Storage Strategy Recap 
Futurestick uses a polyglot persistence model. MySQL stores transactional, relational entities where ACID 
and joins matter — accounts, companies, jobs, applications, interviews, payments. MongoDB stores high-
cardinality, evolving, or document-shaped data — posts, comments, chat, AI artifacts, feed cache, 
interview event timelines. Redis is the fast-path layer for sessions, hot caches, rate limits, queues, and 
pub/sub. Elasticsearch indexes searchable text. All schemas below specify primary keys, foreign keys 
(where applicable), indices, and notes on partitioning.

12. Security & Compliance 
12.1 Threat Model 
Futurestick handles personal data (resumes, salary expectations, interview recordings, identity 
documents), financial data (subscriptions, payouts), and potentially sensitive HR decisions. The threat 
model considers: account takeover (credential stuffing, phishing), privilege escalation (user → admin), 
data exfiltration (insider, leaked tokens, scraping), tampering with hiring decisions, fake recruiters or fake 
candidates (fraud), interview proctoring evasion, denial of service, and supply-chain attacks on 
dependencies. 
12.2 Authentication Hardening 
• Password Policy: Minimum 10 chars, mix recommended, checked against haveibeenpwned k-
anonymity API on signup. 
• Hashing: Argon2id (m=64MB, t=3, p=2) or bcrypt cost 12; pepper stored in KMS. 
• 2FA: TOTP (RFC 6238) with backup codes; WebAuthn for high-trust roles. 
• Suspicious Login Detection: Geo-velocity, device fingerprinting, novel-IP step-up. 
• Session Revocation: Server-side revocation list keyed by JTI; refresh-token reuse triggers cascade. 
• Brute Force: Progressive delay + CAPTCHA + IP block. 
12.3 Authorization 
Every API route declares (1) authentication mode (anonymous / authenticated / company-scoped / 
admin), (2) required permissions, (3) resource ownership rules. The middleware first verifies the JWT, 
then checks permissions in the user's effective set, then evaluates row-level access ("this candidate 
belongs to this company's hiring team"). Permission checks are also re-applied at the data layer: queries 
always include scope filters to prevent leaks via mistakenly broad service code. 
12.4 Data Protection 
• In Transit: TLS 1.3 everywhere, HSTS, certificate pinning for mobile clients, mTLS for inter-service. 
• At Rest: AES-256-GCM via cloud KMS for databases, S3 SSE-KMS for media, envelope encryption for 
sensitive columns (e.g., user phone, identity doc URLs). 
• Field-Level Encryption: PII fields like phone, government ID numbers, and salary expectations are 
encrypted with a per-tenant key. 
• Backups: Encrypted, geographically separated, restoration tested monthly. 
• Secrets: Vault with dynamic database credentials, short-lived; never in env files outside dev.

13. Real-Time Layer & Messaging 
13.1 Socket.IO Topology 
Real-time delivery is centralized in Socket.IO clusters with the Redis adapter. Each Socket.IO node is 
stateless; sticky load balancing is enforced at the gateway via cookie-based affinity. Rooms are 
namespaced per feature: /chat, /feed, /interview, /notifications. Message delivery is at-least-once; 
idempotency on the client de-duplicates by event ID. 
13.2 Notification Pipeline 
Domain Event → notification-svc rules engine → channel selection             → in-app (write 
to MySQL + push via Socket.IO)             → email (BullMQ queue → SES with template)             
→ push (FCM/APNs for installed apps)             → SMS (Twilio for high-priority only) User 
preferences gate channels per category. Digest emails coalesce low-priority notifications. 
13.3 Chat Delivery 
Sending a message follows: client → POST /conversations/:id/messages → chat-svc validates and writes 
to MongoDB → publishes message.created → Socket.IO fan-out to participants in the room. Read receipts 
and typing are ephemeral, never persisted past 30 seconds. End-to-end encryption is opt-in for direct 
chats; the server holds only ciphertext, with key exchange via signal-protocol-style ratchet. 
13.4 Feed Fan-Out 
New posts are pushed to followers using a hybrid model. For authors with under 10,000 followers, the 
post is fanned out into each follower's feed_index at write time. For high-cardinality authors (companies 
with millions of followers, viral users), the post stays in a global by-author index and is pulled into 
followers' feeds at read time. The transition threshold and read-time merge are tunable.

15. Infrastructure & Deployment 
15.1 Environments 
• Local: Docker Compose; isolated MySQL, MongoDB, Redis, MinIO; LocalStack for AWS service 
mocks. 
• Dev: Shared Kubernetes cluster, low-resource, ephemeral databases reset weekly. 
• Staging: Production-mirroring cluster, anonymized production data, full third-party integrations in 
sandbox mode. 
• Production: Multi-AZ, auto-scaling, blue-green deploys for critical services, canaries for risky 
changes. 
15.2 Cluster Layout 
Production runs on Kubernetes (EKS or GKE). Namespaces separate concerns: ingress, app-node, app-
python, ai, data-proxies, observability, security. Each microservice is a Deployment behind a ClusterIP 
Service, exposed externally through the gateway only. Secrets are mounted via External Secrets Operator 
backed by Vault. ConfigMaps hold non-sensitive configuration. Horizontal Pod Autoscalers scale by CPU 
and request rate; AI services additionally scale on queue depth. 
15.3 Datastore Topology 
• MySQL: Primary + 2 read replicas. Daily snapshots, point-in-time recovery via binlog. Schema 
migrations via Prisma Migrate gated by manual approval in production. 
• MongoDB: Replica set (1 primary, 2 secondaries) with automatic failover; sharded clusters for chat 
and feed_index at scale. 
• Redis: Multi-AZ Redis Cluster (6 nodes, 3 shards × replica). Separate clusters for sessions, queues, 
and pub/sub to avoid noisy-neighbor effects. 
• Elasticsearch: 3-node cluster with dedicated master nodes; daily snapshots to S3. 
• S3: Buckets per concern: user-uploads, resumes, recordings, public-assets, backups. Lifecycle rules 
transition cold data to Glacier. 
15.4 CI/CD 
GitHub Actions runs CI: lint, type-check, unit tests, integration tests, dependency scan, secrets scan, and 
build. On merge to main, container images are built (multi-stage Dockerfiles, distroless base) and pushed 
to a private registry. ArgoCD watches a deployment manifest repo and rolls out changes via GitOps. 
Database migrations run as Kubernetes Jobs gated on a successful preflight job that diff-checks the 
proposed schema change.

18. Appendices 
18.1 Glossary 
• ATS: Applicant Tracking System — software used by employers to manage applications. 
• RBAC: Role-Based Access Control. 
• SFU: Selective Forwarding Unit — server that routes WebRTC media to multiple peers. 
• STUN/TURN: Servers that help WebRTC traverse NATs. 
• WebRTC: Browser-native real-time communication standard. 
• SPA: Single-Page Application. 
• SSR / SSG: Server-Side Rendering / Static Site Generation. 
• LCP / CLS / TBT: Core Web Vitals: Largest Contentful Paint, Cumulative Layout Shift, Total Blocking 
Time. 
• PII: Personally Identifiable Information. 
• KMS: Key Management Service. 
• CDC: Change Data Capture — propagating DB changes to other systems. 
• ANN: Approximate Nearest Neighbor search, used for vector similarity. 
• MMR: Maximal Marginal Relevance — balances similarity with diversity in result lists. 
• OWASP: Open Web Application Security Project — security guidelines and tools. 
• SLA / SLO / SLI: Service Level Agreement / Objective / Indicator. 
• DPA: Data Processing Agreement, required under GDPR. 
• MFA / 2FA: Multi-Factor / Two-Factor Authentication. 
• WebAuthn: Public-key-based passwordless authentication standard. 
18.2 Naming Conventions 
• URLs: kebab-case (/job-board, /resume-studio) 
• Database tables: snake_case, plural (users, job_skills) 
• API JSON keys: camelCase 
• Internal events: dot.notation (post.created, application.advanced) 
• Code symbols: PascalCase classes/components, camelCase functions/variables 
18.3 Versioning 
• APIs: /v1, /v2 in URL path; deprecation cycle 12 months 
• Mobile apps: forced upgrade after 6 months; soft prompt earlier 
• Database migrations: forward-compatible, two-phase for breaking changes
